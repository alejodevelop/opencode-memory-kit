# Decisions

Stable repo-level constraints and cross-feature implementation decisions that must outlive the current spec.

Keep durable product direction, target outcomes, and product principles in `product-vision.md` instead.

## How to add entries

- Add only decisions that affect more than one future task or module.
- Use `product-vision.md` for enduring product philosophy or product-level tradeoff rules.
- Keep spec-only or short-lived decisions in the spec instead.
- Prefer one short subsection with affected files.
- Rewrite or remove entries when the decision no longer constrains the current codebase.
- Use Git history for superseded rationale instead of keeping it here.
