# Product Vision

Durable product direction that should stay stable across specs, releases, and implementation phases.

## How to maintain this note

- Keep it short enough to reread before planning or building.
- Capture enduring product intent, not execution detail.
- Update it only when the product's direction or philosophy materially changes.
- Keep roadmap, active tasks, acceptance criteria, rollout plans, and current status in the spec system instead.
- Keep technical constraints in `decisions.md` unless they express a stable product principle.

## Current vision

### Product promise

- In 1-2 sentences, describe the core value the product should reliably deliver.

### Who it serves

- Name the primary user or customer.
- State the core problem, job, or outcome they care about most.

### Desired end state

- List 3-5 durable qualities or capabilities the product should have when it is well built.
- Focus on outcomes, experience, and usefulness more than feature lists.

### Product principles

- List 4-6 stable tradeoff rules that should guide future decisions.
- Prefer short statements that help create, reject, or reshape work.

### Anti-goals

- List 2-4 things the product should avoid becoming.
- Name the kinds of complexity, scope drift, or experience failure that should be resisted.
